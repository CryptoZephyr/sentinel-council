Design notes:
  - trade_count and win_rate are not literal columns in trades.csv (only
    timestamp/symbol/decision/confidence/action/size/balance/pnl/
    explanation exist). Derived both purely from trades.csv: trade_count =
    count of action=="CLOSE" rows; win_rate = % of those CLOSE rows where
    the pnl column (a portfolio-wide running total, not per-trade) shows a
    positive delta from the previous row — avoids any dependency on
    data/portfolio.json, keeping dashboard.py self-contained per the
    "must work with the existing trades.csv structure" requirement.
  - dashboard.py does not import sentinel.py — kept fully standalone so it
    doesn't inherit sentinel.py's module-level side effects (directory
    creation, logging FileHandler setup) and isn't coupled to its Config.
    Symbol list for the "latest decision per symbol" section is derived
    dynamically from whatever symbols appear in trades.csv.
  - The 20-most-recent-trades table omits the explanation column (shown
    separately, in full, in the per-symbol expanders) to keep the table
    readable — explanation text is still read from the CSV and used, just
    rendered in a different section.

Issue #1
  Date:         2026-06-16
  Phase:        Phase 1 (MCP Skill Connection)
  Description:  The five named Bitget Agent Hub Skills (macro-analyst,
                technical-analysis, sentiment-analyst, news-briefing,
                market-intel) do not exist as callable tools on
                bitget-mcp-server (verified via session.list_tools() — 56

                The guessed REST endpoint
                https://api.bitget.com/agent-hub/v1/skills/{name} returns

  Resolution:   Synthesized the five perspectives from real Bitget futures
                market data (ticker, candles, funding rate, open interest)
                via the real MCP tools. news-briefing has no real data
                source and is logged as unavailable every cycle. See
                Decision #1 below.

Issue #2
  Date:         2026-06-16
  Phase:        Post-Phase-7 improvement (explanation persistence)
  Description:  Adding council["explanation"] as a trades.csv column
                surfaced a latent encoding bug: trades.csv was opened with
                Python's platform-default encoding (cp1252 on this Windows
                box), not UTF-8. The em-dash characters in perception text
                wrote as raw cp1252 bytes (0x97), which pandas's default
                UTF-8 reader then failed to decode — confirmed by
                reproducing the exact UnicodeDecodeError. This had been
                latent since Phase 6 but never triggered because no prior
                CSV field contained non-ASCII characters.
  Resolution:   Added encoding="utf-8" to both open() calls in
                _ensure_csv_header() and log_trade_row(). Verified the fix
                by re-running --once and confirming pandas.read_csv()
                parses all rows and the file contains the correct UTF-8
                em-dash bytes (\xe2\x80\x94), not the broken cp1252 byte.

Change #1
  Date:         2026-06-16
  Rule changed: Rule A2 (must call all five Bitget Agent Hub Skills) and
                the call_skill()/MCP-Skill-calling design in
                01_INSTRUCTIONS_FOR_CLAUDE.txt
  Original:     call_skill(skill_name, query) calling named Agent Hub
                Skills (macro-analyst, technical-analysis, etc.) via MCP
                session.call_tool(skill_name, {"query": query}).
  Changed to:   Five named functions (get_macro_signal, get_technical_signal,
                get_sentiment_signal, get_news_signal, get_market_intel_signal)
                that call real Bitget MCP tools (futures_get_ticker,
                futures_get_candles, futures_get_funding_rate,
                futures_get_open_interest) and turn the real numeric data
                into descriptive text, scored by the same keyword
                normalizer the spec requires. news-briefing has no real
                data source and always returns "" with a logged warning
                (covered by Rule A2's stated exception: "Using fewer than
                five is acceptable only if a Skill is genuinely unavailable
                — and this must be logged, not silently skipped").
  Reason:       The five named Skills do not exist as real, callable
                endpoints on either the MCP server or Bitget's REST API
                (confirmed by direct testing — see Issue #1). Building
                against them would mean every cycle silently defaults to
                neutral-50 scores forever, producing a meaningless audit
                trail for judges. User explicitly approved this approach
                over waiting for/inventing fictional Skill responses.
                Recommend confirming with hackathon organizers whether a
                real Agent Hub Skills product exists before final submission.

Change #2
  Date:         2026-06-16
  Rule changed: None of the build rules — internal implementation only.
  Original:     get_technical_signal(), get_sentiment_signal(), and
                get_market_intel_signal() used coarse if/elif buckets,
                each returning one fixed sentence per bucket. BUY_THRESHOLD
                was 75.0.
  Changed to:   All three now compute a continuous real-data signal
                (EMA9/EMA21 gap + RSI for technical; funding rate magnitude
                for sentiment; 24h price change for intel) and pass it
                through a shared _graded_signal_text() helper that mixes a
                calibrated number of bullish/bearish keywords (from the
                existing BULLISH_TERMS/BEARISH_TERMS dicts) proportional to
                signal strength. BUY_THRESHOLD lowered to 70.0 (not the
                originally-proposed 68 — see reason).
  Reason:       Root cause of identical/clustered scores: the keyword
                normalizer's formula (25 + bull/(bull+bear)*50) caps at
                exactly 75 for ANY text containing only one-sided keywords,
                regardless of how many or how strong those keywords are —
                so "strongly positive" and "mildly positive" sentiment text
                scored identically (75) every time. Adding more if/elif
                bands alone would not have fixed this; each band needed a
                deliberately mixed bull/bear keyword ratio. Verified fix
                with python sentinel.py --once run twice: technical/
                sentiment/intel scores now differ meaningfully across
                BTC/ETH/SOL (e.g. intel 49/58/65) where they used to be
                identical or only two-valued. Confidence ceiling under
                current weights (macro and news untouched, both still
                capped at 75/50 respectively) is exactly 72.5, so
                BUY_THRESHOLD=70 leaves a real but reachable 2.5-point
                margin rather than being either unreachable (75) or trivial
                to trip on every cycle (68, which would have caused
                uniform BUY instead of uniform HOLD).
